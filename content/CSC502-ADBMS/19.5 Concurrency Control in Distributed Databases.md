# **Topic: 19.5.1 Locking Protocols in Distributed Databases**

**Goal:** To ensure consistency and serializability when multiple transactions access shared data concurrently in a distributed environment.

**Challenge:**  Extending centralized locking protocols to a distributed setting where data can be replicated and fragmented across multiple sites.

## **Key Concepts:**

*   **Locks:** Mechanisms used to control concurrent access to data items. Common lock modes are shared (S) for reading and exclusive (X) for writing.
*   **Lock Manager:** A component responsible for granting and releasing locks.
*   **Deadlock:** A situation where two or more transactions are blocked indefinitely, each waiting for the other to release a lock.

## **Approaches to Distributed Locking:**

1. **Single Lock-Manager Approach:**
    *   **Concept:** A single designated site acts as the central lock manager for all data items (or in a different version, for all replicas of a given data item).
    *   **Mechanism:**
        *   Transactions send lock requests to the central lock manager.
        *   The lock manager grants or denies locks based on lock compatibility rules.
        *   Transactions can read from any replica but must update all replicas when writing.
    *   **Advantages:**
        *   Simple implementation.
        *   Easy deadlock handling (centralized detection).
    *   **Disadvantages:**
        *   **Single point of failure:** If the lock manager site fails, locking becomes impossible.
        *   **Performance bottleneck:** The lock manager can become a bottleneck, especially with high concurrency.
2. **Distributed Lock Manager:**

    *   **Concept:** The lock management responsibility is distributed among multiple sites.
    *   **Mechanism:**
        *   Each site manages locks for data items stored locally.
        *   For non-replicated data, lock requests are sent to the site where the data resides.
        *   Replication complicates things (see below).
    *   **Advantages:**
        *   Avoids single point of failure.
        *   Better performance potential (parallelism).
    *   **Disadvantages:**
        *   More complex implementation.
        *   Deadlock detection is more difficult (requires global deadlock detection).
3. **Primary Copy:**
    *   **Concept:** One replica of each data item is designated as the primary copy.
    *   **Mechanism:**
        *   Lock requests for a data item are sent to the site that holds the primary copy.
        *   This site manages all locking for that item.
    *   **Advantages:**
        *   Simpler than fully distributed locking.
    *   **Disadvantages:**
        *   The site with the primary copy can become a bottleneck.
        *   If the primary copy site fails, the data item becomes inaccessible.
4. **Majority Protocol:**
    *   **Concept:** A transaction must obtain locks on a majority of the replicas of a data item before proceeding.
    *   **Mechanism:**
        *   Lock requests are sent to multiple sites (more than half).
        *   A transaction can proceed with an operation only if it acquires locks on a majority of the replicas.
        *   Version numbers are used to ensure that reads access the latest version.
    *   **Advantages:**
        *   Can tolerate some site failures.
        *   Decentralized control.
    *   **Disadvantages:**
        *   More complex.
        *   Higher message overhead.
        *   Deadlocks are possible even with a single data item.
5. **Biased Protocol:**
    *   **Concept:** Favors shared (read) locks over exclusive (write) locks to improve read performance.
    *   **Mechanism:**
        *   Shared locks can be obtained from any single replica.
        *   Exclusive locks must be obtained from all replicas.
    *   **Advantages:**
        *   Lower overhead for read operations.
    *   **Disadvantages:**
        *   Higher overhead for write operations.
        *   Still susceptible to deadlocks.
6. **Quorum Consensus Protocol:**

    *   **Concept:** A generalization of majority and biased protocols. Each site is assigned a weight. Read and write operations must acquire a quorum of votes before proceeding.
    *   **Mechanism:**
        *   Each site has a non-negative weight.
        *   `read quorum` (Qr): Minimum weight of sites needed to lock for reading.
        *   `write quorum` (Qw): Minimum weight of sites needed to lock for writing.
        *   `Qr + Qw > S` (where `S` is the total weight of all sites) and `2 * Qw > S` to ensure consistency.
    *   **Advantages:**
        *   Flexible: Can be configured to behave like majority or biased protocols.
        *   Can adjust read/write performance trade-offs.
    *   **Disadvantages:**
        *   More complex to manage.
        *   Deadlocks are possible.

**Handling Replicas:**

*   **Read-one, write-all:** A simple approach where reads can access any replica, but writes must update all replicas. It does not allow for any failures.
*   **Read-one, write-all-available:** An improvement that allows writes to proceed as long as at least one replica is available. Requires mechanisms for reintegrating failed replicas when they recover.

**Deadlock Handling:**

*   **Centralized Deadlock Detection:** Easier with a single lock manager.
*   **Distributed Deadlock Detection:** More complex. Requires sites to exchange information about their local wait-for graphs to detect global deadlocks.

# **Topic: 19.5.2 Timestamping**

*   **Core Idea:** Assign a unique, globally consistent timestamp to each transaction. This timestamp determines the serialization order.  Older transactions (smaller timestamps) conceptually happen "before" younger transactions.
*   **Timestamp Generation:**
    *   **Centralized:** One site generates timestamps.  Simple, but can be a bottleneck and single point of failure.
    *   **Distributed:** Each site generates a *local* unique timestamp (e.g., using a counter or local clock).  Combine this with the site ID to create a *global* unique timestamp.  (e.g., `<local timestamp, site ID>`).
    *   **Logical Clocks:** Ensure timestamps are generated fairly across sites.  If a site receives a message with a timestamp greater than its own, it advances its local clock.
*   **Protocols (using timestamps):**
    *   **Timestamp Ordering Protocol:**  Transactions are executed in timestamp order.  If a transaction tries to access data out of order (e.g., a read with a timestamp older than a previous write), it's aborted and restarted.
    *   **Validation (Optimistic Concurrency Control):**  Transactions proceed without checking.  At commit time, validate that the transaction's reads are still valid based on timestamps.  If validation fails, abort.
*   **Advantages:**
    *   Deadlock-free (transactions don't wait for locks).
    *   Can be implemented without a central coordinator (with distributed timestamp generation).
*   **Disadvantages:**
    *   Can lead to more aborts, especially under high contention.
    *   Requires careful timestamp management.
* **Key Point**: Timestamp defines serialization order.

# **Topic: 19.5.3 Replication with Weak Degrees of Consistency**

*   **Core Idea:** Replicate data across multiple sites, but *don't* enforce strict consistency (ACID properties) across all replicas at all times. This improves availability and performance, but sacrifices immediate consistency.
*   **Trade-off:** Availability and performance vs. consistency.
*   **Two Main Approaches:**

    *   **Master-Slave Replication:**
        *   One site is designated as the "master" for each data item.
        *   *All updates* go to the master.
        *   The master propagates updates to the "slave" replicas.
        *   Reads can be done from *any* replica (master or slave).
        *   **Advantages:**
            *   Simple.
            *   Reads can be very fast (local to a replica).
        *   **Disadvantages:**
            *   The master can be a bottleneck for updates.
            *   Slaves might have slightly outdated data (eventual consistency).
            *   If the master fails, a new master must be elected.
            * Reads from slave might be slightly outdated.
         * **Transaction-Consistent Snapshot:** Slaves must reflect a consistent state of the master as of a specific transaction (all updates from transactions up to a point are applied; updates of later transaction are not applied)
    *   **Multimaster Replication (Update-Anywhere Replication):**
        *   *Any* site can accept updates for any data item.
        *   Updates are propagated to all other replicas.
        *   **Advantages:**
            *   Higher availability for updates (no single master bottleneck).
            *   Updates can be done locally (faster).
        *   **Disadvantages:**
            *   *Much* more complex.
            *   Conflicts are possible (two sites updating the same data item concurrently).
            *   Conflict resolution is required (can be complex, may require manual intervention).
            *   Consistency guarantees are weaker.  Often, only *eventual consistency* is provided.
        *   **Lazy Propagation:** Updates are not propagated as part of the original transaction. They are propagated later, asynchronously.  This improves performance and availability, but further weakens consistency.

# 19.5.4 Deadlock Handling in Distributed Databases

*   **Context:**  Extends deadlock management concepts from centralized systems (Chapter 15) to distributed environments.  The core challenge is that deadlocks can span multiple sites.

## Challenges in Distributed Deadlock Handling

*   **Maintaining the Wait-For Graph:**  The primary difficulty is deciding *how* and *where* to maintain information about which transaction is waiting for which other transaction (the wait-for graph). This is complicated because transactions can access data at different sites.
*   **Global vs. Local Deadlocks:** A deadlock might exist *globally* (involving multiple sites) even if no *local* deadlock exists at any single site.
*   **Communication Delays:**  There are inherent delays in communication between sites.  This means any constructed wait-for graph is an *approximation* of the real, but unknown, state of the system.

## Approaches to Deadlock Handling

We can use either *deadlock prevention* or *deadlock detection* and recovery.  Prevention is often less desirable due to:

*   Unnecessary waiting and rollbacks.
*   Potentially requiring more sites to be involved in a transaction.

Therefore, *deadlock detection* is more commonly used.

### Centralized Deadlock Detection

*   **Mechanism:**
    *   Each site maintains a *local wait-for graph*.  Nodes represent transactions (local or global) that hold or request resources at that site.
    *   A single site is designated as the *deadlock-detection coordinator*.
    *   Sites periodically send their local wait-for graphs to the coordinator.
    *   The coordinator constructs a *global wait-for graph* by combining the local graphs.
    *   The coordinator searches the global graph for cycles.  If a cycle is found, a *victim* transaction is selected for rollback.
    *	The coordinator notifies the sites about the victim. The sites must roll back this victim transaction.

* **Local Wait-for Graph Construction:**
   *	If $T_i$ at site $S_1$ requests a resource held by $T_j$ at site $S_2$:
        1. Send message to site $S_2$.
        2. Insert edge $T_i \rightarrow T_j$ into the local graph of site $S_2$.
   * If a transaction requests a resource at the same site, add edge to the local wait-for graph normally.

*   **Global Graph Construction Triggers:**

    *   Whenever a new edge is added to or removed from a local graph.
    *   Periodically, after a certain number of changes.
    *   Whenever the coordinator needs to check for deadlocks.

*   **Issues and Potential Problems:**

    *   **False Cycles:**  Due to communication delays, the global graph might show a cycle that *doesn't actually exist* in the real system.  This can lead to unnecessary rollbacks.  The PDF provides an example of this (Figure 19.6).
        * For example, a resource release might not be reflected in the global graph immediately, while a new request might already be added, causing a false cycle.

*   **Unnecessary Rollbacks:** Even if a deadlock *did* exist, a transaction involved might have been aborted for an *unrelated* reason.  This can lead to more rollbacks than strictly necessary.

### Distributed Deadlock Detection
The deadlock detection can also happen in a distributed way.
* Several sites are involved in the deadlock detection.
* More complex algorithms.
* More expensive.

## Key Concepts and Definitions

*   **Local Wait-For Graph:**  A graph at each site showing which transactions are waiting for resources held by other transactions *at that site*.
*   **Global Wait-For Graph:**  An approximation of the overall wait-for relationship across *all* sites. Constructed by the deadlock detection coordinator.
*   **False Cycle:** A cycle that appears in the constructed global wait-for graph but *does not* exist in the real, instantaneous state of the distributed system.  Caused by communication delays.
*	**Real graph:** the real but unknown wait-for state.
*	**Constructed graph:** the approximated wait-for graph, generated by the coordinator.
*   **Victim:**  A transaction selected to be rolled back to break a deadlock cycle.
*	**Deadlock-detection coordinator:** a single site, responsible for constructing the global wait-for graph and detecting deadlocks.
## Summary: Advantages and Disadvantages of Centralized Approach

*   **Advantages:**
    *   Relatively simple to implement.
    *   Centralized control makes cycle detection easier.

*   **Disadvantages:**
    *   The coordinator can become a *bottleneck*.
    *   *Vulnerability* - If the coordinator site fails, deadlock detection stops (unless a recovery/election mechanism is in place).
    *   Potential for *false cycles* and unnecessary rollbacks.
